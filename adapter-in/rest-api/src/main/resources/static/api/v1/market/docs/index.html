개요

API 소개

SETOF Commerce Admin API 문서입니다.

항목 설명

Base URL

/api/v1/market

인증

API Gateway에서 JWT 검증 후 X-User-Id, X-User-Roles 헤더 전달

Content-Type

application/json

문서 버전

v1.0.0

인증 및 권한

Gateway 인증 헤더

API Gateway에서 JWT를 검증한 뒤 아래 헤더를 전달합니다.

헤더 필수 설명

X-User-Id

필수

인증된 사용자 ID

X-User-Roles

선택

사용자 역할 (예: ROLE_SUPER_ADMIN)

X-User-Permissions

선택

사용자 권한 목록 (예: *:\*, seller:read)

X-Organization-Id

선택

소속 조직(셀러) ID

권한 레벨

레벨 설명

Public

인증 불필요 (로그인, 토큰 갱신)

인증 필요

유효한 X-User-Id 헤더 필수

셀러 소유자 또는 권한

해당 셀러 소유자이거나 지정된 권한 보유 시 접근 가능

본인 또는 권한

본인이거나 지정된 권한 보유 시 접근 가능

SUPER_ADMIN

ROLE_SUPER_ADMIN 역할 필수

사용 가능한 권한 문자열

개요

상품 그룹 하위의 옵션 구조와 상품(SKU)을 한 번에 수정하는 API입니다.

왜 옵션과 상품을 함께 보내야 하나?

상품(SKU)은 옵션 조합으로 결정됩니다. 예를 들어 "빨강-M" 티셔츠는 색상=빨강, 사이즈=M 옵션의 조합입니다. 옵션이 변경되면(추가/삭제) 상품도 함께 변경되어야 하므로, 두 데이터를 원자적으로 함께 보냅니다.

핵심 개념: selectedOptions

내부 처리 흐름
프론트엔드 요청
  │
  ▼
1. 옵션 그룹 diff 수정 (SellerOptionCommandCoordinator)
   ├─ ID 있음 → 기존 옵션 수정
   ├─ ID null → 신규 옵션 생성 (DB에서 ID 할당)
   └─ 요청에 없는 기존 ID → soft delete
   │
   └─→ resolvedActiveValueIds 반환 (DB 확정 ID 포함)
  │
  ▼
2. 옵션 이름 → ID 매핑 구성
   └─→ optionGroups의 그룹명/값명 + resolvedActiveValueIds로 이름→ID 맵 생성
  │
  ▼
3. 상품(Product) diff 수정 (ProductCommandCoordinator)
   ├─ productId 있음 → 기존 상품 가격/재고/SKU/정렬 수정
   ├─ productId null → 신규 상품 생성 (selectedOptions 이름으로 옵션 매핑)
   └─ 요청에 없는 기존 productId → soft delete

핵심: 옵션이 먼저 DB에 반영된 후, `selectedOptions`의 이름을 사용해 실제 옵션 ID로 resolve합니다. 프론트엔드는 인덱스 계산 없이 이름만 전달하면 됩니다.

function buildSelectedOptions(optionGroups, selectedValues) {
  // 각 옵션 그룹에서 선택한 값의 이름을 그대로 사용
  return selectedValues.map(selected => ({
    optionGroupName: selected.groupName,
    optionValueName: selected.valueName
  }));
}

  • 옵션 값 추가 → 기존 상품 영향 없음

  • 옵션 값 삭제 → 해당 옵션을 사용하는 상품만 제거

  • 옵션 그룹 순서 변경 → 기존 상품 영향 없음
