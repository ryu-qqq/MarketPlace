[[product-update-products-guide]]
==== 상품 일괄 수정 상세 가이드

===== 개요

상품 그룹 하위의 **옵션 구조**와 **상품(SKU)**을 한 번에 수정하는 API입니다.

====== 왜 옵션과 상품을 함께 보내야 하나?

상품(SKU)은 옵션 조합으로 결정됩니다. 예를 들어 "빨강-M" 티셔츠는 색상=빨강, 사이즈=M 옵션의 조합입니다.
옵션이 변경되면(추가/삭제) 상품도 함께 변경되어야 하므로, 두 데이터를 **원자적으로** 함께 보냅니다.


===== 핵심 개념: optionIndices

====== 플랫 인덱스 방식

모든 옵션 그룹의 옵션 값을 **순서대로 1차원 배열로 펼칩니다**. 각 상품은 이 배열의 인덱스로 자신의 옵션 조합을 지정합니다.

[source]
----
optionGroups 배열 순서대로 optionValues를 이어붙인 플랫 리스트:

optionGroups[0].optionValues  →  인덱스 0, 1, ...
optionGroups[1].optionValues  →  인덱스 이어서 계속
----

====== 예시

[source,json]
----
{
  "optionGroups": [
    {
      "optionGroupName": "색상",
      "optionValues": [
        { "optionValueName": "빨강", "sortOrder": 0 },   // → 플랫 인덱스 0
        { "optionValueName": "파랑", "sortOrder": 1 }    // → 플랫 인덱스 1
      ]
    },
    {
      "optionGroupName": "사이즈",
      "optionValues": [
        { "optionValueName": "S", "sortOrder": 0 },      // → 플랫 인덱스 2
        { "optionValueName": "M", "sortOrder": 1 },      // → 플랫 인덱스 3
        { "optionValueName": "L", "sortOrder": 2 }       // → 플랫 인덱스 4
      ]
    }
  ]
}
----

**플랫 리스트**: `[빨강(0), 파랑(1), S(2), M(3), L(4)]`

[cols="1,2,1", options="header"]
|===
| 상품 | 옵션 조합 | optionIndices
| 빨강-S | 빨강 + S | `[0, 2]`
| 빨강-M | 빨강 + M | `[0, 3]`
| 빨강-L | 빨강 + L | `[0, 4]`
| 파랑-S | 파랑 + S | `[1, 2]`
| 파랑-M | 파랑 + M | `[1, 3]`
| 파랑-L | 파랑 + L | `[1, 4]`
|===


===== ID 규칙 (신규 vs 기존)

[cols="1,1,1", options="header"]
|===
| 필드 | null | non-null
| `sellerOptionGroupId` | 신규 옵션 그룹 추가 | 기존 옵션 그룹 수정
| `sellerOptionValueId` | 신규 옵션 값 추가 | 기존 옵션 값 유지/수정
| `productId` | 신규 상품 추가 | 기존 상품 수정
|===

IMPORTANT: 요청에 포함되지 않은 기존 ID는 **자동 삭제(soft delete)** 됩니다. 유지할 항목은 반드시 요청에 포함해야 합니다.

===== 내부 처리 흐름

[source]
----
프론트엔드 요청
  │
  ▼
1. 옵션 그룹 diff 수정 (SellerOptionCommandCoordinator)
   ├─ ID 있음 → 기존 옵션 수정
   ├─ ID null → 신규 옵션 생성 (DB에서 ID 할당)
   └─ 요청에 없는 기존 ID → soft delete
   │
   └─→ resolvedActiveValueIds 반환 (DB 확정 ID 포함, 플랫 리스트)
  │
  ▼
2. 상품(Product) diff 수정 (ProductCommandCoordinator)
   ├─ productId 있음 → 기존 상품 가격/재고/SKU/정렬 수정
   ├─ productId null → 신규 상품 생성 (optionIndices로 옵션 매핑)
   └─ 요청에 없는 기존 productId → soft delete
----

**핵심**: 옵션이 먼저 DB에 반영되어 실제 ID가 확정된 후, `resolvedActiveValueIds` 플랫 리스트가 만들어집니다. `optionIndices`는 이 플랫 리스트의 인덱스입니다.


===== 프론트엔드 구현 가이드

====== optionIndices 계산 로직 (JavaScript)

[source,javascript]
----
function calculateOptionIndices(optionGroups, selectedValues) {
  // 1. 플랫 리스트 만들기
  const flatValues = optionGroups.flatMap(group => group.optionValues);

  // 2. 선택된 옵션 값의 인덱스 찾기
  return selectedValues.map(selected => {
    const index = flatValues.findIndex(v =>
      v.optionValueName === selected.optionValueName &&
      v === selected  // 또는 ID로 비교
    );
    return index;
  });
}

// 예시: 색상(빨강,파랑) + 사이즈(S,M,L)에서 "파랑-M" 상품
// flatValues = [빨강, 파랑, S, M, L]
// selectedValues = [파랑, M]
// → optionIndices = [1, 3]
----

====== 인덱스 재계산이 필요한 경우

옵션 값의 **순서나 개수가 변경**될 때마다 모든 상품의 `optionIndices`를 재계산해야 합니다:

* 옵션 값 추가/삭제
* 옵션 그룹 순서 변경
* 옵션 그룹 추가/삭제

TIP: 프론트엔드에서 옵션 조합 테이블 UI를 보여줄 때, 내부적으로 각 셀이 어떤 플랫 인덱스를 가지는지 관리하면 됩니다.


===== 필드 상세

[cols="2,1,1,4", options="header"]
|===
| 필드 | 타입 | 필수 | 설명

4+h| optionGroups
| `sellerOptionGroupId` | Long | - | 기존 그룹 ID (null이면 신규)
| `optionGroupName` | String | O | 옵션 그룹명 (예: "색상", "사이즈")
| `canonicalOptionGroupId` | Long | - | 표준 옵션 그룹 ID (매핑용)

4+h| optionGroups.optionValues
| `sellerOptionValueId` | Long | - | 기존 값 ID (null이면 신규)
| `optionValueName` | String | O | 옵션 값명 (예: "빨강", "M")
| `canonicalOptionValueId` | Long | - | 표준 옵션 값 ID (매핑용)
| `sortOrder` | Integer | O | 정렬 순서 (0부터)

4+h| products
| `productId` | Long | - | 기존 상품 ID (null이면 신규)
| `skuCode` | String | O | SKU 코드
| `regularPrice` | Integer | O | 정가 (원)
| `currentPrice` | Integer | O | 판매가 (정가 이하)
| `stockQuantity` | Integer | O | 재고 수량 (0 이상)
| `sortOrder` | Integer | O | 정렬 순서 (0부터)
| `optionIndices` | List<Integer> | O | 플랫 옵션 값 인덱스 목록
|===


===== 주의사항

. **optionIndices 범위**: 전체 옵션 값 개수를 초과하면 안 됩니다
. **상품별 optionIndices 개수**: 옵션 그룹 수와 동일해야 합니다 (그룹 2개면 인덱스 2개)
